# Requirements Document

## Introduction

This feature adds Two-Factor Authentication (2FA) via TOTP (Time-based One-Time Password) to SYNCRO, a financial subscription management application. Users can optionally enroll a TOTP authenticator app (e.g., Google Authenticator, Authy) to protect their accounts. Team owners can enforce 2FA for all team members. The implementation leverages Supabase Auth MFA APIs.

## Glossary

- **SYNCRO**: The financial subscription management application.
- **User**: An authenticated SYNCRO account holder.
- **Team_Owner**: A User with administrative privileges over a team in SYNCRO.
- **TOTP**: Time-based One-Time Password — a 6-digit code generated by an authenticator app per RFC 6238.
- **Authenticator_App**: A third-party TOTP application such as Google Authenticator or Authy.
- **MFA_Service**: The Supabase Auth MFA subsystem used to enroll, challenge, and verify TOTP factors.
- **Enrollment_Flow**: The UI and API sequence through which a User registers a TOTP factor.
- **Recovery_Code**: A one-time-use alphanumeric code that allows a User to bypass TOTP verification when access to the Authenticator_App is lost.
- **Security_Settings_Page**: The `/settings/security` page where Users manage their 2FA configuration.
- **2FA_Verification_Page**: The `/auth/2fa` page shown after password authentication when 2FA is required.
- **AAL1**: Authenticator Assurance Level 1 — session authenticated with password only.
- **AAL2**: Authenticator Assurance Level 2 — session additionally verified with a second factor.
- **QR_Code**: A scannable image encoding the TOTP secret URI, displayed during enrollment.

## Requirements

### Requirement 1: TOTP Enrollment

**User Story:** As a User, I want to enroll a TOTP authenticator app, so that I can protect my account with a second factor.

#### Acceptance Criteria

1. WHEN a User initiates 2FA setup on the Security_Settings_Page, THE MFA_Service SHALL generate a TOTP secret and return a QR_Code URI and a plain-text secret.
2. WHEN the MFA_Service returns enrollment data, THE Enrollment_Flow SHALL display the QR_Code and the plain-text secret to the User simultaneously.
3. WHEN a User submits a 6-digit TOTP code during enrollment, THE MFA_Service SHALL verify the code against the enrolled factor using `challengeAndVerify`.
4. IF the submitted TOTP code is invalid or expired, THEN THE Enrollment_Flow SHALL display an error message and allow the User to re-enter the code without restarting enrollment.
5. WHEN enrollment verification succeeds, THE Enrollment_Flow SHALL mark the User's 2FA status as enabled and persist it to the database.

---

### Requirement 2: Recovery Code Generation

**User Story:** As a User, I want to receive recovery codes when I enable 2FA, so that I can regain access if I lose my authenticator device.

#### Acceptance Criteria

1. WHEN 2FA enrollment verification succeeds, THE SYNCRO SHALL generate exactly 10 unique one-time-use recovery codes for the User.
2. WHEN recovery codes are generated, THE SYNCRO SHALL store a bcrypt hash of each recovery code in the database, not the plain-text value.
3. WHEN recovery codes are generated, THE Enrollment_Flow SHALL display all 10 plain-text recovery codes to the User exactly once.
4. THE Enrollment_Flow SHALL provide a mechanism for the User to download the 10 recovery codes as a plain-text file before dismissing the enrollment confirmation screen.
5. WHEN a recovery code has been used once, THE SYNCRO SHALL invalidate that recovery code so it cannot be reused.

---

### Requirement 3: 2FA Verification on Login

**User Story:** As a User with 2FA enabled, I want to be prompted for my TOTP code after entering my password, so that my account remains protected even if my password is compromised.

#### Acceptance Criteria

1. WHEN a User completes password authentication and the session assurance level is AAL1 with a next level of AAL2, THE SYNCRO SHALL redirect the User to the 2FA_Verification_Page before granting access to the application.
2. WHEN a User submits a 6-digit TOTP code on the 2FA_Verification_Page, THE MFA_Service SHALL verify the code and elevate the session to AAL2.
3. IF the submitted TOTP code is invalid or expired on the 2FA_Verification_Page, THEN THE 2FA_Verification_Page SHALL display an error message and allow the User to re-enter the code.
4. WHEN a User submits a valid recovery code on the 2FA_Verification_Page, THE SYNCRO SHALL verify the code against stored hashes, invalidate the used code, and elevate the session to AAL2.
5. WHEN the session is elevated to AAL2, THE SYNCRO SHALL redirect the User to the originally requested route.

---

### Requirement 4: 2FA Status and Management

**User Story:** As a User, I want to view and manage my 2FA configuration from the security settings page, so that I have full control over my account security.

#### Acceptance Criteria

1. THE Security_Settings_Page SHALL display whether 2FA is currently enabled or disabled for the User's account.
2. WHEN 2FA is enabled, THE Security_Settings_Page SHALL display the date on which 2FA was enabled.
3. WHEN a User requests to disable 2FA on the Security_Settings_Page, THE SYNCRO SHALL require the User to enter a valid TOTP code or recovery code before disabling.
4. WHEN the User provides a valid TOTP code or recovery code to disable 2FA, THE MFA_Service SHALL unenroll the TOTP factor and THE SYNCRO SHALL remove all associated recovery codes from the database.
5. IF a Team_Owner has enforced 2FA for the team, THEN THE Security_Settings_Page SHALL prevent the User from disabling 2FA and SHALL display a message indicating that 2FA is required by the team.

---

### Requirement 5: Team-Level 2FA Enforcement

**User Story:** As a Team_Owner, I want to require 2FA for all team members, so that I can enforce a minimum security standard across my team.

#### Acceptance Criteria

1. THE Security_Settings_Page SHALL provide a Team_Owner with a toggle to require 2FA for all members of the team.
2. WHEN a Team_Owner enables the 2FA requirement, THE SYNCRO SHALL record the enforcement policy against the team in the database.
3. WHEN a team member without 2FA enabled logs in and the team's 2FA enforcement policy is active, THE SYNCRO SHALL redirect the User to the Enrollment_Flow before granting access to the application.
4. WHEN a team member with 2FA already enabled logs in and the team's 2FA enforcement policy is active, THE SYNCRO SHALL apply the standard AAL1-to-AAL2 verification flow defined in Requirement 3.
5. WHEN a Team_Owner disables the 2FA requirement, THE SYNCRO SHALL update the enforcement policy and SHALL allow team members to disable their own 2FA.

---

### Requirement 6: Security and Session Integrity

**User Story:** As a User, I want my 2FA session state to be handled securely, so that my account cannot be accessed through session manipulation.

#### Acceptance Criteria

1. WHILE a session is at AAL1 and the User's account has 2FA enabled, THE SYNCRO SHALL restrict access to all authenticated routes until the session is elevated to AAL2.
2. WHEN a TOTP verification attempt fails 5 consecutive times within a 10-minute window, THE SYNCRO SHALL temporarily lock the 2FA verification for that session for 15 minutes and SHALL display a lockout message to the User.
3. THE SYNCRO SHALL transmit all TOTP codes and recovery codes exclusively over HTTPS.
4. WHEN a User successfully enrolls 2FA or disables 2FA, THE SYNCRO SHALL send a confirmation email to the User's registered email address.
